üéØ What is DIRK.g?

DIRK.g is the Business Analyst and Solutions Architect component of the DIRK system, powered by Google's Gemini AI. It specializes in analyzing business requirements, designing solution architectures, and creating detailed specifications that DIRK.c can implement.

Key Capabilities

  • Business Requirements Analysis - Stakeholder analysis, acceptance criteria, business rules
  • Solution Architecture Design - Technology selection, design patterns, integration strategies
  • Risk Assessment - Security analysis, performance considerations, scalability planning
  • Implementation Planning - Development phases, resource requirements, timelines
  • DIRK Principles Application - Systematic application of all 8 cognitive principles

üìã DIRK.g Commands Reference

dirk_g_ba

Business Analysis Command

dirk_g_ba "requirement description"

Analyzes business requirements and generates comprehensive specifications including stakeholder analysis, functional requirements, acceptance criteria, and risk assessment.

Example:
dirk_g_ba "user authentication system with OAuth2 and multi-factor authentication"

dirk_g_sa

Solution Architecture Command

dirk_g_sa "architectural requirement"

Designs solution architecture with technology recommendations, design patterns, security considerations, and implementation strategies.

Example:
dirk_g_sa "scalable rate limiting system with Redis backend for API protection"

dirk_g_workflow

Complete Analysis Workflow

dirk_g_workflow "complete requirement"

Runs complete Business Analysis ‚Üí Solution Architecture workflow, generating comprehensive specifications ready for DIRK.c implementation.

Example:
dirk_g_workflow "real-time web crawler monitoring dashboard with performance analytics"

dirk_g_test

System Test Command

dirk_g_test

Tests DIRK.g integration and displays available commands with usage examples.

üß† DIRK Principles in Business Analysis

DIRK.g applies all 8 core DIRK principles systematically to ensure high-quality analysis and architecture:

1. Systematic Doubt

In BA: Challenge business assumptions, question unstated requirements, validate stakeholder needs.

In SA: Question technology choices, challenge performance assumptions, validate architecture decisions.

2. Foundational Reasoning

In BA: Anchor requirements to business objectives, ensure alignment with enterprise strategy.

In SA: Base architecture on proven patterns, align with enterprise architecture principles.

3. Formal Logic

In BA: Ensure requirements are complete, consistent, and testable with clear acceptance criteria.

In SA: Apply proven design patterns, ensure logical component interactions.

4. Empirical Grounding

In BA: Validate requirements against real user needs and market research.

In SA: Base technology decisions on performance metrics and real-world constraints.

5. Abstraction

In BA: Break complex business needs into manageable, understandable components.

In SA: Design clean interfaces, clear separation of concerns, modular architecture.

6. Inference to Best Explanation

In BA: Identify the real business problem behind stated requirements.

In SA: Select optimal technology and architecture patterns with clear rationale.

7. Understanding Limits

In BA: Acknowledge scope, budget, timeline, and resource constraints.

In SA: Document technical constraints, performance limits, and trade-offs.

8. Cognitive Awareness

In BA: Consider stakeholder perspectives, communication clarity, change management.

In SA: Focus on long-term maintainability, team capabilities, operational complexity.

üí° Best Practices & Tips

Tip 1: Be specific and detailed in your requirement descriptions. The more context you provide, the better the analysis.
Tip 2: Use dirk_g_workflow for complete features. Use dirk_g_ba and dirk_g_sa separately for focused analysis.
Tip 3: Review the generated specifications before handing off to DIRK.c to ensure they match your vision.
Note: DIRK.g requires Gemini CLI to be installed and configured. Make sure you have API access.

Common Usage Patterns

  • New Feature Development: Use dirk_g_workflow for complete analysis
  • Architecture Reviews: Use dirk_g_sa to evaluate existing designs
  • Requirements Clarification: Use dirk_g_ba to refine unclear requirements
  • Technical Decisions: Use dirk_g_sa to compare technology options
